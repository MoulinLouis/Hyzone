# Review: 2026-03-17 Pickaxe Upgrades

Target: `docs/plans/2026-03-17-pickaxe-upgrades.md`

## Findings

- High: The plan does not cover the full persistence and UI wiring for six new upgrade types, and the current code will break if `MineUpgradeType` gains more than one extra value. The plan only calls out new `mine_players` columns and enum extension (`docs/plans/2026-03-17-pickaxe-upgrades.md:235-248`, `:298-301`), but `MinePlayerStore` currently only loads and saves `bag_capacity_level` (`hyvexa-parkour-ascend/src/main/java/io/hyvexa/ascend/mine/data/MinePlayerStore.java:89-99`, `:194-208`). On top of that, `MinePage` still hardcodes one upgrade color/name entry while iterating `MineUpgradeType.values()` (`hyvexa-parkour-ascend/src/main/java/io/hyvexa/ascend/mine/ui/MinePage.java:65-70`, `:216-243`), so adding a second upgrade will throw `ArrayIndexOutOfBoundsException`.

- High: Fortune and the AoE upgrades promise overflow auto-sell, but that conflicts with the current bag-capacity gate in both mining systems. The plan says Fortune replaces `blocksGained = 1` before inventory add (`docs/plans/2026-03-17-pickaxe-upgrades.md:61`) and says Jackhammer/Stomp/Blast should respect bag capacity with excess auto-sold (`:78`, `:114`, `:149`). Today both `MineBreakSystem` and `MineDamageSystem` abort before the block is even claimed if `canAddToInventory(blocksGained)` is false (`hyvexa-parkour-ascend/src/main/java/io/hyvexa/ascend/mine/system/MineBreakSystem.java:106-111`, `:132-139`; `hyvexa-parkour-ascend/src/main/java/io/hyvexa/ascend/mine/system/MineDamageSystem.java:93-98`, `:131-139`). If `blocksGained` becomes 2/3, or AoE logic reuses the same guard, valid breaks near a full bag will be rejected instead of partially stored and auto-sold.

- Medium: Haste is designed around a `MovementSpeed` component that does not appear to be a real or supported path in this repo. The plan explicitly says to apply/remove a `MovementSpeed` component on mine enter/exit (`docs/plans/2026-03-17-pickaxe-upgrades.md:176-184`, `:318`), but the local API notes say walk speed changes are not handled through a simple component and that even `horizontalSpeedMultiplier` is reset every tick; the documented workaround is modifying `maxHorizontalSpeed` directly (`docs/HYTALE_API.md:110-115`). I also could not find any existing `MovementSpeed` usage in the codebase. This needs an implementation spike before it should be treated as a low-risk upgrade.

- Medium: Momentum feedback is under-specified on the HUD side. The plan requires a combo counter and timer bar in the mine HUD (`docs/plans/2026-03-17-pickaxe-upgrades.md:29`, `:320`), but the UI section only mentions extending `MinePage` with upgrade cards (`:296-301`). The current `MineHudManager` only updates crystals, inventory, cooldowns, block health, and toasts (`hyvexa-parkour-ascend/src/main/java/io/hyvexa/ascend/mine/hud/MineHudManager.java:60-145`, `:148-193`, `:247-321`), and the current HUD file only exposes those existing sections. Without explicit `Ascend_MineHud.ui` changes plus a new periodic combo/timer update path, Momentum cannot deliver the feedback loop the design depends on.

- Medium: The shared AoE break utility contradicts the stated rule that multi-HP blocks must be skipped. Jackhammer, Stomp, and Blast all say multi-HP blocks are skipped (`docs/plans/2026-03-17-pickaxe-upgrades.md:79`, `:113`, `:148`), but the sample `breakBlocksAt(...)` utility just claims and breaks positions directly (`:276-285`). In the current implementation, multi-HP behavior is enforced through `BlockDamageTracker.recordHit(...)` and the break/no-break branch in `MineDamageSystem` (`hyvexa-parkour-ascend/src/main/java/io/hyvexa/ascend/mine/system/MineDamageSystem.java:100-115`). If the new AoE helper does not explicitly consult block HP before breaking, it will bypass durability and insta-break blocks that should require repeated hits.
